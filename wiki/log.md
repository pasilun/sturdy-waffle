## [2026-04-28] session | concept: per-feature model tuning via eval
- Pages: [[llm-eval-as-dev-canary]]
- Notes: Extended the eval-as-dev-canary page with a "Tuning model choice per feature" section. The strongest use of the harness in a multi-call pipeline: each feature has a distinct success signal (extraction → validator binary; mapping → per-fixture accuracy ratio), and the boundary fixture (`lunch.pdf` for staff-meals/entertainment) is where a cheaper model first overclaims confidence on a wrong answer — overconfidence is worse than under-confidence because it misleads the accountant's "do I need to look closely" cue. Documented a 5-cell experiment matrix (status quo / cheap-extract / smart-map / bottom / top) intended as one constant change + `./gradlew :api:eval` per cell. Three caveats specific to model selection: (1) three fixtures is statistically thin, add foreign-currency / multi-line / sloppy-scan before trusting cheaper; (2) non-determinism — same fixture+model varies across runs, mitigate with N≥3 runs per cell or `temperature: 0`; (3) hardcoded model constants force edit+recompile per swap, lifting to `application.yml` (the deferred /simplify item) is the prerequisite for scripting a real sweep. Section also frames eval and prod-DB as complementary observational studies — eval = fast pre-ship signal across a tiny matrix, persisted `model`/`prompt_version` columns = slow long-tail signal across the real distribution — and notes that dynamic model escalation (low-confidence mapping retries on a stronger model) is now near-term feasible since Phase 3 persistence has landed; the eval harness is where you'd find the threshold first.
